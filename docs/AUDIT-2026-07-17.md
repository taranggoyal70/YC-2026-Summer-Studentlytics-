# Product Audit — 2026-07-17

Goal: fully built product, nothing hardcoded or local, everything real and functional end to end. This audit catalogues every gap between that goal and the current code, and records the decided target version (see ADR-0002, ADR-0003, ADR-0004).

## What is already real

- **Auth**: Clerk end-to-end — `ClerkProvider` in the frontend, JWKS verification, role checks, rate limiting, and audit logging in `backend/security.py`. Env-driven, no hardcoded keys.
- **Opportunities**: real CRUD through FastAPI with validated payloads and role checks (storage is still a JSON file — see below).
- **Roster CRUD**: real API calls from `src/services/api.ts` (storage is still a JSON file).
- **Analysis pipeline**: genuine face recognition + Whisper transcription over uploaded recordings in `backend/main.py` — real processing, local-only runtime.
- **Config discipline**: `VITE_API_ENDPOINT` fails loudly in production instead of silently falling back — good.

## Blocking gaps

1. **Fabricated domain data in product paths.** `src/data/semesterData.ts` hand-writes ~10 fictional students across four semesters; `src/data/transformStudents.ts` re-exports them as `realStudents`. Consumed by Sessions, Analytics, Attendance, AttendanceTracking, Cohort, StudentProfile, Leaderboard pages. SessionsPage synthesizes "workshops" with hardcoded Oct/Nov 2025 dates. → Delete both files; render API data with empty states (ADR-0004).
2. **No Session/Space/Organization entities.** The API exposes only `/students`, `/opportunities`, `/videos`. A presence-intelligence product needs sessions as first-class records. → Rebuild API around the CONTEXT.md nouns.
3. **Local-only storage.** `students.json`, `opportunities.json`, in-memory `processing_jobs`, `face_encodings.json` on disk. State dies on redeploy and forks with >1 worker. → Managed Postgres + object storage; jobs as DB rows (ADR-0003).
4. **Biometric data committed to git.** `backend/data/face_encodings.json`, ~47 face crops in `backend/data/extracted_faces/`, and `backend/data/users.db` are tracked in a public repo — contradicts the product's own privacy commitments. → Untracked in this audit; **history purge still required** (force-push, owner's call).
5. **Dual auth stacks.** Legacy SQLite/custom-JWT auth (`backend/auth.py`, `main_auth.py`) coexists with Clerk. → Delete legacy stack (ADR-0002).
6. **Dead AWS configuration.** `src/config/aws-config.ts` names an S3 bucket, Rekognition collection, and Lambda that no code calls; `videoService.ts` talks to the local FastAPI. Contradicts ADR-0001. → Delete.
7. **False status documentation.** `FUNCTIONALITY_STATUS.md` and the `*_TASKS_COMPLETED.md` files claimed "AWS integration 100% WORKING" and "ALL MOCK DATA REPLACED WITH REAL DATA" while every page rendered fabricated arrays. → Removed in this audit.
8. **Presence model too weak for the glossary.** Analysis results store one `checkInAt`/`checkOutAt` pair plus a `returnedAfterLeave` flag; CONTEXT.md requires multiple presence windows as evidence, and the full attendance-decision enum (absent, late, left early, unknown) is missing. → `presence_windows` array becomes source of truth.

## Decided target version

Vercel-hosted Vite frontend (Clerk auth) → containerized FastAPI backend (Railway/Fly/Render class) → managed Postgres + S3-compatible object storage. API nouns follow CONTEXT.md: Organization → Space → Session → Recording → Analysis Job → Presence Windows → Attendance Decision. No fabricated data anywhere; demo content only via a seed script writing through the real API.

## Inputs required before implementation can go live

- Backend host choice + account (Railway / Fly / Render)
- `DATABASE_URL` (managed Postgres) and object-storage credentials
- Clerk production instance: `VITE_CLERK_PUBLISHABLE_KEY`, `CLERK_ISSUER`, `CLERK_JWKS_URL`
- `VITE_API_ENDPOINT` set in Vercel once the backend URL exists
- Decision on purging biometric files from git history (requires force-push)
